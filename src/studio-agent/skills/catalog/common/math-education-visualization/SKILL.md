# 数学教育可视化

这个技能不是教你“把图做得更花”，而是教你怎样让学生更快看懂数学关系。

默认目标只有四个：

- 重点能被迅速看到
- 无关信息尽量少
- 图像、符号、逻辑能对上
- 结构、过程、变化能被跟住

如果用户只说“帮我做一个数学动画 / 数学图示 / 数学课件页面”，不要直接进入风格设计。先按下面的顺序工作。

## 标准流程

### 第一步：先判定教学任务

先把任务压缩成一句话：

- 学生这一步到底要看什么？
- 他现在最容易看错什么？
- 这张图是为了看结构、看步骤、看变化、看对比，还是看数量关系？

可以直接把任务归入以下五类之一：

- `看结构`：看组成关系、位置关系、整体框架
- `看步骤`：看推导顺序、构造顺序、算法顺序
- `看变化`：看参数变化、极限逼近、连续运动
- `看对比`：看两类对象、两种情况、两种结果的差异
- `看数量`：看大小、比例、增减、斜率、长度、面积

如果用户的描述同时包含多项，强制选一个“当前主任务”。不要试图用一张图同时承担所有解释责任。

### 第二步：选呈现模式

先决定内容如何分布在空间和时间上，再设计样式。

可选模式只有三类：

- `静态呈现`：单图、并列多图、一个页面内的多子图
- `伪动态呈现`：分步图、关键帧、多阶段排布、轨迹暗示
- `动态呈现`：真实动画、参数连续变化、交互滑块驱动变化

用下面这套决策：

1. 如果核心是“让学生停下来比较和检查”，优先 `静态呈现`
2. 如果核心是“让学生按顺序跟步骤”，优先 `伪动态呈现`
3. 如果核心是“连续变化本身就是知识点”，优先 `动态呈现`

更具体地说：

- 讲立体几何的多视图、函数图像对比、分类讨论，优先静态
- 讲代数推导、作图过程、证明步骤，优先伪动态
- 讲参数变化、极限逼近、轨迹形成、形态变换，优先动态

如果拿不准，默认不要上动画。动画一旦用错，代价比静态图大得多。

需要细分判断时，读 [references/presentation-modes.md](references/presentation-modes.md)。

### 第三步：给画面对象分层

不要一上来问“用什么颜色”。先问：

- 哪个对象是当前研究对象
- 哪些对象只是辅助说明
- 哪些对象只是背景框架
- 哪些对象其实可以删掉

把对象分成四层：

- `研究对象`：学生此刻必须追踪的主体
- `辅助对象`：帮助说明关系，但不是主角
- `背景对象`：只负责提供方向感和上下文
- `噪音对象`：删掉后不损失理解，甚至更清楚

默认处理规则：

- 研究对象保留完整可读性
- 辅助对象降低存在感，但不能消失
- 背景对象用弱化样式
- 噪音对象直接删除

典型噪音：

- 过多文字说明
- 密集网格线和细碎刻度
- 过大的坐标轴箭头
- 重复标注同一个关系
- 此刻不再是主角、却仍然用高权重显示的旧对象

需要更系统的判断时，读 [references/visual-variables.md](references/visual-variables.md)。

### 第四步：把数学信息映射成视觉变量

先判断你现在表达的信息本质上是哪一类：

- `定性信息`：分类、区分、谁是谁
- `顺序信息`：先后、强弱、阶段、递进
- `定量信息`：大小、比例、倍数、长度、面积、位置差异

然后按默认映射：

- 定性信息：优先用 `色相`，其次用 `形状/线型`
- 顺序信息：优先用 `明度/透明度/尺寸`
- 定量信息：优先用 `位置/长度/面积/尺寸`

不要这样做：

- 不要用不同颜色的色相去表达“大小顺序”
- 不要用形状去表达“数值高低”
- 不要用深浅颜色去表达精确倍数关系
- 不要在平等对象之间随意改大小和明暗，否则会误导“哪个更重要”

一句话记忆：

- 分类靠颜色
- 递进靠深浅或大小
- 数量靠位置和长度

### 第五步：最后再上颜色

颜色只能在结构已经稳定后介入，而且每一个时刻最好只承担一个主要任务。

颜色任务分三类：

- `展示`：稳定地区分对象，适合长时间阅读
- `突出`：把注意力强行拉到关键点
- `动态演变`：让学生看懂对象正在变化

默认规则：

- 背景和非重点对象尽量中性化
- 同一语义对象保持同色对应
- 展示场景降低饱和度，并保持同层对象明度一致
- 突出场景提高饱和度，但不要把明度推得太高
- 动态演变场景要分清“质变”和“量变”

特别注意：

- 如果是“同一个对象越来越接近目标”，保持同一色相，只改饱和度或透明度
- 如果是“数学定义本身跨越了类别”，再考虑用色相跃迁
- 不要把“质变”和“量变”的色彩逻辑混在一起

需要默认色库和 HSL 起点时，读 [references/color-guidelines.md](references/color-guidelines.md)。

## 常见任务模板

### 模板 1：做一张讲解图

输出时至少给出这六项：

1. 教学目标
2. 选用的呈现模式
3. 研究对象 / 辅助对象 / 背景对象
4. 每类对象用什么图形元素承载
5. 视觉变量怎么分配
6. 颜色怎么服务理解

### 模板 2：做一段数学动画

除了上面六项，再补四项：

1. 哪些内容必须动，哪些必须静
2. 动的是量变还是质变
3. 镜头或时间轴如何分段
4. 哪一段最容易造成认知超载

### 模板 3：审查一张已有图

按以下顺序批评，不要只说“好看不好看”：

1. 这张图的教学目标是否清晰
2. 呈现模式是否和知识点匹配
3. 主次对象是否分清
4. 是否存在明显视觉噪音
5. 视觉变量是否和信息类型匹配
6. 颜色是在帮助理解，还是在制造干扰

## 输出格式要求

当你实际使用这个技能产出方案时，优先输出这种结构：

### 1. 教学任务判断

- 当前主任务是什么
- 次任务是什么
- 学生可能卡在哪里

### 2. 呈现模式选择

- 选什么模式
- 为什么不用另外两种

### 3. 画面对象分层

- 研究对象
- 辅助对象
- 背景对象
- 删除项

### 4. 视觉编码方案

- 每个对象用什么图形元素
- 每个数学关系用什么视觉变量

### 5. 色彩方案

- 这是展示 / 突出 / 动态演变中的哪一种
- 主色怎么分配
- 哪些地方必须保持中性

### 6. 风险检查

- 哪些地方可能造成误读
- 哪些元素还可以继续减法

## 失败检查清单

如果你发现方案有下列问题，说明你做偏了：

- 颜色先于结构被决定
- 一张图同时承载太多教学任务
- 研究对象和背景对象一样抢眼
- 用颜色深浅假装表达精确数量
- 动画只是“动起来了”，但没有解释价值
- 图中文字比图形本身更忙
- 学生必须靠读说明才能看懂图意
